# Vimax Web 项目宪章

## 核心原则

### 一、代码质量优先（不可妥协）

代码质量是所有技术决策的基石，任何开发活动必须以此为前提。

- **可读性**：代码即文档。命名必须语义化，函数职责单一，复杂度可控。禁止超过 200 行的函数和超过 500 行的单文件模块。
- **一致性**：全项目统一使用 ESLint + Prettier 强制代码风格，禁止绕过格式化检查提交代码。
- **健壮性**：所有外部输入必须校验与净化；所有异步操作必须有错误处理；所有 Promise 必须有 `.catch()` 或 try-catch 包裹。
- **可维护性**：每个模块必须有清晰的职责边界。共享逻辑提取为独立的工具函数或 Hook，杜绝复制粘贴。
- **禁止条款**：禁止提交 `console.log` 调试代码；禁止硬编码魔法数字和字符串；禁止遗留被注释掉的废弃代码。

### 二、测试标准（不可妥协）

测试不是可选项，而是交付标准的重要组成部分。

- **单元测试**：核心业务逻辑、工具函数、自定义 Hook 必须有对应的单元测试，目标覆盖率 ≥ 80%。
- **组件测试**：每个可复用的 UI 组件必须包含渲染测试、交互测试和边界状态测试。
- **集成测试**：关键用户流程（登录、核心业务操作、支付等）必须有端到端的集成测试覆盖。
- **测试先行**：Bug 修复必须先编写能复现问题的测试用例，确认测试失败后再修复，确保测试通过后方可合并。
- **质量门禁**：CI 流水线中测试失败视为构建失败，禁止合并到主分支。

### 三、用户体验一致性

用户在产品的任何触点上获得统一、流畅、可预期的体验。

- **设计系统**：所有 UI 元素必须使用统一的设计系统/组件库，禁止临时拼凑样式。颜色、间距、字体、圆角、阴影等设计 Token 必须从主题变量中引用。
- **交互规范**：按钮、表单、导航、弹窗等交互模式必须在全站保持一致。相同功能在不同页面的操作路径和视觉呈现必须统一。
- **响应式设计**：所有页面必须适配移动端（≥375px）、平板（≥768px）和桌面端（≥1280px）三种断点。
- **可访问性（a11y）**：所有交互元素必须支持键盘导航；图片必须有 alt 文本；表单必须有正确的 label 关联；色彩对比度满足 WCAG 2.1 AA 标准。
- **加载与空状态**：每个异步加载区域必须有 Loading 骨架屏或加载指示器；空数据状态必须有友好的引导文案和操作入口；错误状态必须有明确的错误信息和重试机制。
- **国际化就绪**：所有面向用户的文案必须使用 i18n 方案管理，禁止在组件中硬编码中文或任何语言的文本字符串。

### 四、性能要求

性能直接影响用户留存率和转化率，必须以量化指标驱动优化。

- **加载性能**：
  - 首屏加载（FCP）≤ 1.5 秒，最大内容绘制（LCP）≤ 2.5 秒
  - 路由切换页面渲染 ≤ 300ms
  - 首屏 JavaScript 体积 ≤ 200KB（gzip 后）
- **运行时性能**：
  - 用户交互响应（INP）≤ 200ms
  - 列表渲染 ≥ 1000 条时强制使用虚拟滚动
  - 禁止在渲染路径中进行复杂计算，必须使用 `useMemo` / `computed` 缓存
- **资源优化**：
  - 所有图片必须使用现代格式（WebP/AVIF），并提供合适的尺寸变体和懒加载
  - 字体文件必须子集化（subset）并预加载
  - 第三方脚本必须异步加载或延迟加载，不得阻塞首屏渲染
- **监控与度量**：
  - 生产环境必须接入性能监控平台，实时追踪 Core Web Vitals
  - 每次发布必须对比关键性能指标，性能回退 >10% 视为发布阻塞项

## 技术决策与实施治理

### 技术选型原则

- **成熟优先**：优先选择社区活跃、文档完善、有长期支持（LTS）的技术栈和依赖库。
- **简约至上**：遵循 YAGNI（You Aren't Gonna Need It），不被过度工程化所困。新增依赖必须经过评审，说明必要性和替代方案。
- **渐进增强**：核心功能不依赖最新浏览器 API，确保主流浏览器兼容性不低于近 2 个主要版本。

### 代码审查强制项

以下项目在代码审查中必须逐一核验，任一不通过则驳回：

1. 代码风格是否符合 ESLint/Prettier 规则（自动化检查）
2. 单元测试是否覆盖新增/修改的核心逻辑
3. 是否存在硬编码文本（应使用 i18n）
4. 是否存在性能隐患（不必要的重渲染、未缓存的计算、未虚拟化的大列表）
5. UI 是否使用设计系统组件，样式是否引用主题 Token
6. 错误处理和边界条件是否完备

### 架构决策记录（ADR）

以下类型的技术决策必须记录为 ADR（Architecture Decision Record）并存入 `docs/adr/` 目录：

- 引入新的第三方依赖或更换现有核心依赖
- 对项目架构、目录结构或模块划分的重大调整
- 状态管理策略变更
- 路由设计或权限模型变更

### 例外与豁免机制

- 若因技术债务或紧急需求需豁免某条原则，必须在 PR 中明确标注 `CONSTITUTION_EXCEPTION:` 前缀，说明原因、影响范围和后续偿还计划。
- 所有豁免必须获得至少一位技术负责人的书面批准。
- 豁免事项需录入技术债务登记表，并在 30 个自然日内完成偿还。

### 修订流程

- 任何人对宪章的修订提议需以 Issue 形式提出，阐明修订内容、理由和影响评估。
- 修订内容需经团队核心成员讨论并达成共识后方可生效。
- 每次修订后版本号递增，修订记录追加到版本历史中。
