**功能分支**: `001-vimax-web-platform`

**创建日期**: 2026-06-17

**状态**: Draft

**输入**: 用户描述："创建基于 ViMax: Agentic Video Generation 的 Web 管理平台，将 CLI 调用转为 Web 调用，包括项目管理、可视化运行、运行结果记录、人工操作记录、统计分析"

## Clarifications

### Session 2026-06-17

- Q: Web 平台的访问控制？ → A: 无需登录认证，创建用户表预设默认用户 "muze"（ID=1），操作记录绑定 user_id 预留多用户扩展
- Q: 技术栈选择？ → A: 前端 React + Ant Design 6，后端 Python 3.13 + FastAPI 0.137，前后端分离
- Q: 数据存储策略？ → A: SQLite 存储所有元数据，文件系统存储生成产物，统计查询直接在 SQLite 通过 SQL 聚合完成
- Q: ViMax 流水线调用方式？ → A: 运行 main_idea2video.py 从头执行，通过修改 interrupt_step 控制断点，kill subprocess 强制中断
- Q: 项目模板机制？ → A: 2-3 个内置模板（含 working_dir 骨架 + YAML + config.py），创建时复制到项目，配置双向同步（编辑器 ↔ working_dir）
- Q: ComfyUI 不可用时的步骤状态处理？ → A: 区分"完全完成"和"部分完成"，部分完成可进入下一步但必须重新执行该步骤
- Q: 浏览器关闭/切换项目时 ViMax 进程的行为？ → A: 不影响运行，仅用户中断或 FastAPI 关闭才终止，全局最多一个项目生成中
- Q: ViMax stdout 日志如何持久化？ → A: 临时日志文件，每次启动覆盖写入，运行期间持续追加，WebSocket/SSE 实时推送

## 用户场景与测试 *(必填)*

### 用户故事 1 — 创建并管理视频生成项目 (优先级: P1)

作为一名创作者，我希望通过 Web 界面创建新的视频生成项目，选择一个内置模板来初始化项目配置（包括 working_dir、模型 API 配置、执行模式等），并能管理所有历史项目，以便我可以系统化地组织每次视频创作，而不需要手动编辑 YAML 配置文件。

**优先级理由**: 这是平台的入口功能，没有项目管理，所有后续的生成、查看、分析都无法进行。创作者的所有工作都围绕"项目"展开。

**独立测试**: 可通过创建一个新项目、查看项目列表、切换项目来完全验证，交付"项目组织能力"这一核心价值。

**验收场景**:

1. **Given** 用户在项目列表页，**When** 点击"新建项目"，选择模板并填写项目名称和创意描述，**Then** 系统在 working_dir 下创建对应的项目目录，并在界面显示新项目卡片。
2. **Given** 用户有一个已创建的项目，**When** 用户点击进入该项目，**Then** 可以看到项目的配置信息（working_dir、模型选择、中断步骤、执行模式）和当前执行状态。
3. **Given** 用户存在多个项目，**When** 用户在项目列表中浏览，**Then** 可以按创建时间、状态筛选和搜索项目。
4. **Given** 用户需要修改项目配置，**When** 用户在项目设置中修改 idea2video.yaml 和 config.py 中的参数，**Then** 修改后的配置在下次执行时生效。

### 用户故事 2 — 分步执行并实时观察视频生成过程 (优先级: P1)

作为一名创作者，我希望在 Web 界面中启动视频生成后，系统能按照 ViMax 的流水线步骤（story → character → portrait → script → storyboard → shot → scene）逐步执行，每步完成后暂停等待我确认，并且我能看到终端输出的实时日志，以便我能掌控生成节奏并在关键节点介入调整。

**优先级理由**: 这是 Web 平台区别于 CLI 的核心能力——可视化、可控的分步执行。创作者需要能"看到"AI 在做什么，并在不满意时及时干预而不是等整个流程跑完才发现问题。

**独立测试**: 可通过启动一个项目的生成流程，观察每一步的暂停确认弹窗、终端日志实时滚动、确认后进入下一步的完整链路来独立验证。

**验收场景**:

1. **Given** 项目的 interrupt_step 配置为 "story"，**When** 用户点击"开始生成"并完成 story 步骤生成，**Then** 系统暂停并弹出确认框，展示 story.txt 的内容，等待用户确认是否继续。
2. **Given** 某步骤正在执行中，**When** 用户查看执行面板，**Then** 可以看到该步骤的终端 stdout 实时滚动输出，类似终端体验。
3. **Given** 某步骤执行完成且用户确认通过，**When** 系统进入下一步骤，**Then** 上一步的输出文件（如 story.txt、script.json、storyboard.json）自动出现在项目的文件浏览器中。
4. **Given** 用户在步骤确认时看到不满意的结果，**When** 用户选择"重新生成"，**Then** 系统重新执行当前步骤，覆盖之前的输出文件。
5. **Given** 终端 stdout 正在滚动显示，**When** 页面刷新或用户切换页面后返回，**Then** 之前的 stdout 内容不再保留（实时展示，不持久化存储）。

### 用户故事 3 — 查看和编辑生成过程的输出文件 (优先级: P2)

作为一名创作者，我希望在 Web 界面中直接查看和编辑 ViMax 生成的中间文件，包括故事文本（story.txt）、剧本（script.json）、分镜（storyboard.json）等，以便我可以手动修正 AI 生成的不理想内容，提升最终视频质量。

**优先级理由**: 创作者经常需要微调 AI 输出，直接编辑能力让平台不止是"观察者"而是"协作工具"。但核心的生成+可视化流程比编辑更重要。

**独立测试**: 可通过打开项目中已生成的 story.txt 并编辑保存、查看 script.json 的结构化内容、修改 storyboard 中某个镜头的描述来独立验证。

**验收场景**:

1. **Given** 项目的 story 步骤已完成，**When** 用户点击文件列表中的 story.txt，**Then** 系统在编辑器中展示故事内容，支持编辑和保存。
2. **Given** script.json 已生成，**When** 用户打开该文件，**Then** 系统以结构化方式展示所有场景的剧本内容，支持编辑每个场景的文本。
3. **Given** storyboard.json 已生成，**When** 用户查看分镜内容，**Then** 可以看到每个镜头的可视化描述和关联的参考信息。
4. **Given** 用户编辑并保存了某个中间文件，**When** 后续步骤执行时，**Then** 系统使用编辑后的文件内容作为输入。

### 用户故事 4 — ComfyUI 图像/视频结果管理与重试 (优先级: P2)

作为一名创作者，我希望在 Web 界面中查看 ComfyUI 生成的图像和视频结果，能够将每个生成结果与对应的 workflow 和 prompt_id 关联起来，并且可以编辑 workflow 参数后重新生成（抽卡模式），对满意的结果进行确认，未确认的结果系统会提醒。

**优先级理由**: ComfyUI 生成具有不确定性，创作者需要多次"抽卡"才能得到满意效果。管理这些生成结果、追溯生成参数、标记确认状态是提升视频质量的关键工具。

**独立测试**: 可通过查看某镜头的 ComfyUI 生成结果列表、修改 workflow 参数后重新生成、确认某个结果、查看是否有未确认结果的提醒来独立验证。

**验收场景**:

1. **Given** 某个镜头通过 ComfyUI 生成了 3 张首帧图像，**When** 用户查看该镜头的生成结果，**Then** 可以看到所有 3 张图像，每张关联显示对应的 workflow 和 prompt_id。
2. **Given** 用户对某张生成图像不满意，**When** 用户打开该图像的 workflow 编辑器并修改提示词参数后点击"重试"，**Then** 系统发起新的 ComfyUI 生成任务，生成完成后新结果追加到结果列表中。
3. **Given** 存在 5 个生成结果其中 2 个已确认、3 个未确认，**When** 用户浏览项目，**Then** 未确认的结果区域有醒目的提醒标识（如角标数字和颜色提示）。
4. **Given** 用户确认了某个生成结果，**When** 查看结果列表，**Then** 已确认的结果自动收起折叠，仅展示第一张缩略图，未确认的结果保持展开。
5. **Given** 某个 ComfyUI 生成任务，**When** 任务完成，**Then** 记录本次生成的耗时和类型（首帧/尾帧/视频）。

**验收场景**:

### 用户故事 6 — 生成数据统计与成功率分析 (优先级: P3)

作为一名创作者，我希望平台能统计和展示每次视频生成项目的关键数据（场景数、文件数量、文件大小、各步骤耗时），分析成功与失败的原因，并追踪哪些修改操作能够提升生成成功率，以便我不断优化创作流程并为未来的智能优化提供数据基础。

**优先级理由**: 数据分析是平台长期价值的关键——帮助创作者理解"什么配置和操作方式成功率更高"，也为后续 LLM/RL 优化提供训练数据。但在 MVP 阶段，核心生成功能优先。

**独立测试**: 可通过查看项目维度的统计数据（场景数、文件量、耗时分布）、全局维度的成功率趋势、以及"某类型修改 → 成功率提升"的关联分析来独立验证。

**验收场景**:

1. **Given** 一个已完成的项目，**When** 查看该项目的统计面板，**Then** 可以看到场景总数、各类文件数量和大小、总生成耗时、各步骤耗时分布。
2. **Given** 某个 ComfyUI 生成任务失败，**When** 查看统计报告，**Then** 可以看到失败记录包含失败原因（如超时、API 错误、图片校验未通过等）。
3. **Given** 多个项目已完成，**When** 查看全局统计仪表盘，**Then** 可以查看整体成功率、失败率以及按时间段的变化趋势。
4. **Given** 记录了多次"编辑配置/文件后重新生成 → 成功"的模式，**When** 查看改进分析页面，**Then** 可以追踪特定修改操作与成功率提升之间的关联关系。

### 边界情况

- 当 ComfyUI 服务不可用时（健康检查失败），系统如何处理生成请求？——ComfyUI 不可用不影响 ViMax 流水线运行（等同于 comfyui_enable=false）。但该步骤标记为"部分完成"状态，允许进入下一步骤。下次运行时必须从该步骤重新执行，直到 ComfyUI 结果生成成功才标记为"完全完成"。
- 当某个步骤执行时间超长（如大型 scene 的视频渲染），用户在此期间关闭了浏览器或切换到其他项目怎么办？——ViMax 进程在服务端持续运行，不受浏览器关闭或项目切换影响。仅用户手动中断（kill）或 FastAPI 服务关闭时 ViMax 进程才会终止。用户重新打开页面后可恢复查看当前执行状态和 stdout。系统同时限制全局最多只有一个项目处于"生成中"状态。
- 多个用户同时操作同一个项目时如何避免冲突？——MVP 阶段假设单用户操作，依赖文件层面的覆盖即可。
- 当 working_dir 下已有部分生成文件时，如何判断是否可从断点继续？——系统应沿用 ViMax 的"检查文件是否存在则跳过"的逻辑，在 Web 层展示"已缓存"状态标记。
- 当某个步骤因 ComfyUI 不可用而标记为"部分完成"后，用户后续如何处理？——该步骤在下次 ViMax 运行时自动重新执行（清除部分缓存并将 interrupt_step 设为该步骤），直到 ComfyUI 生成成功，步骤升级为"完全完成"。

## 需求 *(必填)*

### 功能需求

**项目管理**
- **FR-001**: 系统必须支持创建新的视频生成项目，每个项目关联一个唯一的 ViMax working_dir 目录。
- **FR-002**: 系统必须提供 2-3 个内置项目模板（如"标准视频"、"快速预览"、"高质量"），模板包含 working_dir 目录结构、idea2video.yaml 和 config.py 的默认配置。创建项目时，将模板的 YAML 和 config.py 同步复制到项目的 working_dir 下。
- **FR-003**: 系统必须允许用户在 Web 界面中阅读和修改项目的 idea2video.yaml 和 config.py 配置文件，每次保存时自动同步备份到项目的 working_dir。当用户切换到不同项目时，系统自动加载该项目的现存配置到编辑器中，保证每个项目的配置独立沿用。
- **FR-004**: 系统必须支持项目列表的浏览、搜索和按状态筛选（待生成、生成中、已完成、失败）。
- **FR-005**: 系统必须能够删除项目及其关联的 working_dir 目录。

**分步执行与可视化**
- **FR-006**: 系统必须按照 ViMax 流水线的步骤顺序执行，支持的步骤包括：story、character、portrait、script、environment、environment_images、storyboard、shot_description、camera_tree、camera_frame、shot_video、shot_transition、scene_video、scene_transition 以及 hanzi 模式下的特有步骤。
- **FR-007**: 系统必须在每次启动 ViMax 前，根据用户确认的下一步骤自动修改 idea2video.yaml 中的 interrupt_step 参数，通过此机制控制流水线停止位置。
- **FR-008**: 系统必须在每个步骤暂停时展示当前步骤的输出摘要（文本文件展示内容，图片/视频文件展示缩略图），并提供"确认继续"和"重新生成"两个操作选项。
- **FR-009**: 系统必须在步骤执行期间实时展示 ViMax 进程的 stdout 输出流。stdout 通过临时文件实现持久化——每次 ViMax 启动时覆盖写入临时文件，运行期间持续追加，Web 端通过 WebSocket/SSE 实时读取推送。临时文件在下次运行时被覆盖。
- **FR-010**: 系统必须支持手动触发项目中已执行步骤的重新生成（清除该步骤的缓存文件后将 interrupt_step 设为该步骤，重新运行 ViMax）。同时支持通过 kill subprocess 强制中断正在运行的 ViMax 进程。
- **FR-011**: 系统必须限制全局最多只有一个项目处于"生成中"状态。浏览器关闭或切换项目不影响 ViMax 进程运行，仅用户手动中断或 FastAPI 服务关闭才终止进程。
- **FR-012**: 系统必须区分步骤的两种完成状态：**"完全完成"**（该步骤所有产出均已生成，包括 ComfyUI 图像/视频）和 **"部分完成"**（仅文本类产出生成，ComfyUI 部分因服务不可用而跳过）。部分完成的步骤允许进入下一步，但下次运行时必须从该步骤重新执行直到完全完成。

**文件查看与编辑**
- **FR-013**: 系统必须以文件树的形式展示项目 working_dir 下的所有生成文件。
- **FR-014**: 系统必须支持在线查看文本类文件（story.txt、script.json、storyboard.json 等），对 JSON 文件进行结构化展示。
- **FR-015**: 系统必须支持在线编辑文本类文件并保存，保存后的内容应影响后续步骤的执行输入。

**ComfyUI 结果管理**
- **FR-016**: 系统必须为每个 ComfyUI 生成结果关联以下元数据：prompt_id、生成类型（首帧/尾帧/视频）、生成耗时。workflow 定义（ui 版和 api 版）存储于 `working_dir/workflows` 目录，前端展示和重试时从该目录读取，不在数据库中冗余存储。
- **FR-017**: 系统必须支持查看所有 ComfyUI 生成结果的图片/视频预览。
- **FR-018**: 系统必须支持编辑 workflow 参数后重新提交生成任务（抽卡/重试模式）。
- **FR-019**: 系统必须支持对生成结果进行"确认"操作，确认后的结果自动收起折叠。
- **FR-020**: 系统必须在项目中存在未确认结果时展示提醒标识。

**操作记录**
- **FR-021**: 系统必须自动记录以下类型的人工操作：编辑文件、确认生成结果、拒绝生成结果、触发重新生成。所有操作记录绑定 user_id，MVP 阶段默认绑定到预设用户 "muze"（ID=1）。
- **FR-022**: 每条操作记录必须包含时间戳、操作类型、操作对象、变更摘要、关联用户 ID。

**统计分析**
- **FR-023**: 系统必须提供项目维度的统计摘要：场景数量、各类文件数量和大小、各步骤生成耗时、ComfyUI 生成成功/失败次数。
- **FR-024**: 系统必须记录每次 ComfyUI 生成失败的原因（超时、API 错误、图片校验未通过、连接失败等）。
- **FR-025**: 系统必须提供全局维度的成功率和失败率统计，支持按时间段筛选。
- **FR-026**: 系统必须追踪配置修改和文件编辑操作与后续生成成功率之间的关联关系，为未来的 LLM/RL 学习优化提供数据基础。

### 关键实体

- **项目（Project）**: 代表一次视频生成任务。包含名称、创建时间、状态、关联的 working_dir 路径、配置文件快照、执行的模板类型。
- **执行步骤（Step）**: 项目流水线中的一个环节。包含步骤名称、状态（待执行/执行中/完全完成/部分完成/失败）、开始时间、结束时间、输出文件列表。部分完成表示文本类产出已生成但 ComfyUI 生成未完成，允许进入下一步但下次运行须重新执行。
- **ComfyUI 生成结果（Generation Result）**: 某步骤中 ComfyUI 的一次生成输出。包含文件路径、prompt_id、类型、耗时、确认状态。workflow 定义从 working_dir/workflows/ 文件系统读取
- **操作记录（Operation Log）**: 人工在 Web 界面上执行的一次干预。包含时间、类型、对象、摘要。
- **项目模板（Template）**: 预设的项目配置方案。包含 working_dir 目录骨架、默认的 idea2video.yaml 和 config.py 内容。创建项目时模板配置被复制到项目的 working_dir，后续项目的配置修改通过双向同步机制（Web 编辑器 ↔ working_dir 文件系统）保持一致性。
- **用户（User）**: 操作归属主体。MVP 阶段预设唯一用户 "muze"（ID=1），无需登录认证，所有项目、操作记录和生成结果的 user_id 默认关联到此用户。

## 成功标准 *(必填)*

### 可衡量成果

- **SC-001**: 创作者能在 3 分钟内完成一个项目的创建和配置，包括选择模板和填写创意描述。
- **SC-002**: 创作者通过分步执行能完整跟踪一个视频生成的全部步骤，每一步的终端日志在暂停时清晰可见，步骤间过渡等待不超过 2 秒。
- **SC-003**: 创作者能在不离开 Web 界面的情况下完成对 story.txt、script.json 等中间文件的查看和编辑，编辑保存后内容在 3 秒内反映到文件系统。
- **SC-004**: ComfyUI 生成结果页面能清晰展示每个结果的预览、关联信息（prompt_id、耗时）和确认状态，未确认结果有肉眼可辨识的提醒。
- **SC-005**: 90% 的 ComfyUI 生成任务能在操作记录中准确追溯到编辑操作和配置变更。
- **SC-006**: 项目的统计数据（场景数、文件量、耗时分布）在项目完成后 5 秒内完成计算并展示。
- **SC-007**: 全局成功率统计能按日和周两个粒度展示趋势变化，数据滞后不超过 1 分钟。

## 假设

- ViMax 源代码位于 `/Users/wangchangbin/ai/ViMax-main`，Web 平台通过 subprocess 调用 `main_idea2video.py` 启动流水线。由于 ViMax 必须从头运行，分步控制通过以下机制实现：每次运行前修改 `interrupt_step` 配置指定断点，ViMax 从头启动后自动跳过已缓存的步骤（文件存在则跳过），运行至新断点处停止。Web 端可通过 `subprocess.kill()` 强制中断正在运行的 ViMax 进程。
- ComfyUI 服务地址已在项目配置中指定，Web 平台默认复用该地址进行健康检查和结果管理。
- 工作目录 `/Users/wangchangbin/ai/vimax_ru` 作为参考，项目创建时的 working_dir 将在此路径下的子目录或用户指定的新路径中。
- MVP 阶段无需登录认证，本地单用户使用（localhost 访问），项目数据存储在本地文件系统。但系统仍需创建用户表并预设默认用户 "muze"（ID=1），所有操作记录和生成结果均绑定到此默认用户，为后续多用户扩展预留数据模型。
- 项目的 idea2video.yaml 和 config.py 通过 Web 编辑器直接修改，每次保存后同步写入 working_dir 下的实际文件。切换项目时从 working_dir 回读当前配置到编辑器，保证配置随项目独立沿用。
- ViMax 运行日志通过临时文件实现：每次启动 ViMax 时覆盖写入项目 working_dir 下的临时日志文件，运行期间持续追加，Web 端通过 WebSocket/SSE 从该文件实时读取并推送。文件在下次启动时被覆盖，不长期存储历史日志。
- ViMax 流水线的步骤定义（story、character、portrait 等）保持与 ViMax 源码中的 interrupt_step 定义一致，Web 层不做额外映射。
- 统计分析的"修改 → 成功率"关联，在 MVP 阶段以简单关联展示为主，高级的 LLM/RL 学习优化留作后续迭代。
- 技术栈：前端使用 React + Ant Design 6（SPA 架构），后端使用 Python 3.13 + FastAPI 0.137，前后端分离。FastAPI 通过 subprocess 调用 ViMax Python 流水线。实时 stdout 通过 WebSocket/SSE 推送。
- 数据存储：SQLite 存储所有元数据（项目、步骤、ComfyUI 结果、操作记录、用户、统计数据），文件系统仅存储生成产物（story.txt、图片、视频等）。统计查询通过 SQL 聚合直接在 SQLite 完成，MVP 阶段无需独立分析数据库。
